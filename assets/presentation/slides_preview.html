What is IRONGRID AI?

The modern conversational interface replacing repetitive front desk inquiries

Grounded Front-Desk Operations

Designed to answer visitor queries regarding memberships, equipment, class schedules, and rules. Unlike generic LLM assistants, IRONGRID AI restricts its responses to facts pre-verified and indexed in its ChromaDB knowledge base.

  • ✓ Eliminates front-desk phone queues and visitor wait times
  • ✓ Enforces strict boundaries against medical/workout planning
  • ✓ Sub-second response latencies with persistent indexing
  • ✓ Contextual session memory tracking across 10 chat turns
🔍 Semantic Indexing
Maps natural user queries into dense 384-dimensional arrays using SentenceTransformers.
🛡️ Strict Guardrails
System prompt constraints block off-topic chat or health advisory claims.
⚙️ Robust Error Handling
Gracefully serves direct text templates automatically if external APIs fail.
📊 Metadata Analytics
Surfaces confidence ratings and category labels along with responses.

Slide 5

Why RAG?

Comparison: Traditional LLM Memory vs. Retrieval-Augmented Generation

❌ TRADITIONAL CHATBOTS (LLM Memory)

  • • Knowledge Source: Static pre-trained weights (cannot adapt to changes)
  • • Hallucination Risk: High (generates false values and details confidently)
  • • Attribution: None (unable to explain or reference information origin)
  • • Updates: Costly (requires expensive model fine-tuning or retraining)
  • • Behavior Boundaries: Weak (prone to giving unsolicited medical/workout tips)

✅ IRONGRID AI (RAG Core)

  • • Knowledge Source: Live ChromaDB persistent vector index chunks
  • • Hallucination Risk: Guarded (strictly locked to retrieved facts)
  • • Attribution: Explicit (returns source subcategory tags and relevance values)
  • • Updates: Instant (simple database JSON file edits & re-embedding)
  • • Behavior Boundaries: Strong (locked strictly to reception assistant role)

System Architecture

End-to-end RAG data processing pipeline (Double-row flow)

1. 👤 User Query
Visitor inputs a natural language inquiry.
→
2. 🖥️ Chat UI
JavaScript packages parameters to fetch POST.
→
3. ⚡ Flask API

→
4. 🤖 Embedder

→
5. 🗄️ ChromaDB
Cosine similarity identifies matrix matches.
↓
6. 🔍 Top-5 Search
Pulls top 5 relevant facts from local DB.
→
7. 📄 Prompt Build
Assembles context facts and 10-turn history.
→
8. ♊ Gemini API
Sends bounded content for text synthesis.
→
9. 🛡️ Grounded text
Compiles answers matching the constraints.
→
10. 📈 UI Display
Displays reply, confidence rating and sources.

Slide 10

Technical Challenges & Solutions

Implementation workarounds resolving key runtime issues

⚠️ LLM API Quota Exhaustion
Issue: Gemini free tier throws 429 exceptions under rapid concurrent inquiries.
Solution: Implemented direct local fallback to serve raw text template records.
📁 Vector DB Path Resolution
Issue: Folder paths fail when testing across different directories or OS environments.
Solution: Dynamically resolved absolute DB paths relative to app script folder.
🛡️ Out-of-Bounds Queries
Issue: Users prompt the receptionist to generate diet templates or medical claims.
Solution: Prompt bounds enforce early rejection, keeping replies focused on basic facts.
⏳ API Request Latency
Issue: Slow network retry parameters freeze the browser chat panel for up to 90 seconds.
Solution: Set short client timeouts with early cancel parameters, triggering local fallback immediately.
